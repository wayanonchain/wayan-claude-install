What you will build

Wayan Agents is a reusable Telegram AI agent framework. You will end up with two personal AI agents living on your own server 24/7, reachable from Telegram: Jupiter — the primary research/content/analysis agent (tokens, PDFs, voice notes, videos, posts), and Uran — an independent operations/server/support agent that reads logs, diagnoses the box, and restarts Jupiter when it falls over. Plus long-term semantic memory (OpenViking), a knowledge base in Markdown + Git + Obsidian, and a one-command installer with profession presets.

Section 2

What Wayan Agents are

Wayan Agents is a kit for building personal AI agents. You rent a small server, run the installer from an open GitHub repository — and get two agents in Telegram. Every part of the system does one understandable job:

PartRole in plain words
JupiterThe main agent: research, analysis, content. The one you message every day.
UranThe backup "mechanic": server diagnostics, logs, restarting Jupiter when it crashes. A separate bot — an independent line of communication.
TelegramThe interface. No websites or consoles in daily life — you just message a bot (text, voice, files, video).
VPSThe always-on rented server where the agents live 24/7 — even when your laptop is off.
Claude CodeThe "brain": the Claude model with tools — it thinks, runs commands, reads and writes files.
GitHubThe source of truth: code, templates, rules, and the full history of every change. Where you install from and commit to.
OpenVikingSemantic long-term memory: the agent finds relevant knowledge by meaning, not by exact keywords.
MarkdownThe format of all knowledge: plain text files that humans and agents read equally well.
ObsidianThe human-facing window into that Markdown knowledge: read, edit, organize.
GroqSpeech-to-text (Whisper): voice notes and video audio tracks become text.
OpenAIThe embeddings/VLM provider for OpenViking — what makes the memory "semantic".
ffmpegExtracts keyframes from videos so Claude can literally see the footage.
WAYAN_PACKA profession preset at install time: onchain analyst, creator, devops, student, founder.
Why two agents instead of one? A single agent is a single point of failure: if it hangs, nobody can revive it remotely. Uran is an independent second channel with its own bot and its own service, holding scoped sudo for exactly one job: "read the logs and restart Jupiter". Jupiter goes down → you message Uran → Uran fixes it.
Section 3

Architecture: three diagrams

3.1 The main flow: from message to knowledge

User ↓ Telegram (text · voice · files · video) ↓ Jupiter / Uran (gateway + task queue) ↓ Claude Code (the brain: thinks and acts) ↓ Skills · Rules · Memory · OpenViking ↓ Markdown → Git → Obsidian

Every message travels this path: Telegram receives it, the gateway queues a task, Claude Code solves it using skills, rules, and memory — and the result settles into Markdown files that Git versions and Obsidian displays.

3.2 The video flow: visual analysis

Telegram video ↓ download into a temporary folder ↓ Groq: audio → transcript + ffmpeg: keyframes └──────────────┬───────────────┘ ↓ Claude: looks at frames + reads transcript ↓ one merged review: ## Audio / ## Visual / ## Combined ↓ Markdown transcript + Telegram reply

Videos are analyzed on two channels at once: Groq transcribes the sound, ffmpeg extracts a few evenly-spaced frames, Claude inspects each frame and merges both signals into one answer. Frames and the raw video are temporary: after the reply, only the Markdown transcript remains.

3.3 The update flow: repo-first deploy

GitHub repository (source of truth) ↓ scripts/update.sh (git pull + tests + drift check) ↓ tests pass ↓ scripts/deploy-gateway.sh (backup → sync) ↓ /opt/wayan-jupiter + /opt/wayan-uran ↓ systemd restart (service restart with verification)

Code is never edited directly on the server. Change → commit to GitHub → update script with tests → deploy with an automatic backup. Details in section 16.

Section 4

Where everything lives

After installation, a handful of paths matter on the VPS. Understanding them is important: half of all beginner problems are "edited the wrong file in the wrong place".

PathWhat it is
/root/wayan-claude-installThe GitHub repo cloned onto the server. The working copy of the source of truth: updates and deploys start here.
/opt/wayan-jupiter/gatewayProduction code for Jupiter: the Telegram gateway + Python environment. Code only arrives here via the deploy script.
/opt/wayan-uran/gatewayThe same for Uran — a separate copy, so the agents stay independent.
/home/wayan/.claude-lab/jupiterJupiter's workspace: CLAUDE.md (the role), skills/, orchestration/, transcripts/, uploads/, PACK.md (if a pack was chosen).
/home/wayan/.claude-lab/uranUran's workspace — same structure.
/home/wayan/.openvikingOpenViking memory: the ov.conf config (holds keys, permissions 600) and the index data. Never enters Git.
/etc/wayan-jupiter.envJupiter's secrets and settings: bot token, Groq key, voice/video flags. Permissions 600/640.
/etc/wayan-uran.envThe same for Uran — separate token, separate file.
/opt/wayan_pirat_botA legacy bot, not part of this repository. Leave it alone: it has its own life and its own timers.

Repository vs production: why they are different places

The repository (/root/wayan-claude-install) is the "blueprints": code with history, tests, and a way back. Production (/opt/wayan-*) is the "running machine": what systemd services actually execute.

Why GitHub is the source of truth. When truth lives in Git, any machine can be rebuilt from scratch, every change is visible in history, and tests run before code reaches production.

Why editing /opt directly is dangerous. Such an edit is invisible to Git ("drift"): the next deploy silently overwrites it, there is nothing to roll back to, and "it worked yesterday" becomes unanswerable. This project had a real drift incident once — which is why deploy-gateway.sh --check now exists to catch it.

Section 6

Beginner glossary

33 terms you'll meet in this guide. Each block expands: what it is, why it matters, an analogy, the common mistake, and an example from Wayan Agents.

SSH

What it is: a protocol for securely connecting to a remote computer through a terminal.

Why: the whole server is managed over SSH — installing, configuring, reading logs.

Analogy: a teleport into the server room: you "sit at its keyboard" without leaving your chair.

Common mistake: losing track of where you are — local terminal or already on the server. Look at the prompt: root@server = you're on the server.

In Wayan: ssh root@SERVER_IP is the first step of the install.

VPS

What it is: a Virtual Private Server — a rented virtual machine in a datacenter.

Why: the agents must run 24/7, and your laptop doesn't.

Analogy: a rented apartment for your programs: small, but always "home".

Common mistake: picking 512 MB RAM to save money — the memory layer and agents will suffocate. Get 2 GB.

In Wayan: the whole system lives on one VPS running Ubuntu 22.04/24.04.

Linux user

What it is: an account inside the server with its own files and permissions.

Why: separation of powers: what a program running under this name is allowed to do.

Analogy: office badges — the janitor's badge and the director's badge open different doors.

Common mistake: doing everything as root because "it's simpler". That way one mistake levels the whole server.

In Wayan: the installer creates the wayan user — the agents run as it, not as root.

root

What it is: the Linux superuser — can do anything, no questions asked.

Why: installing packages and creating services has to be done as root (via sudo).

Analogy: the master key to the entire building.

Common mistake: running things as root that should run as a regular user (like the Claude login).

In Wayan: install.sh requires root; but you log in to Claude as wayan: sudo -u wayan -i.

Git

What it is: a version control system — it remembers every state of your files.

Why: change history, rollback, collaboration, the "source of truth".

Analogy: save points in a game: you can always return to a previous save.

Common mistake: editing files on the server outside Git — the changes get lost on the next deploy.

In Wayan: all code and templates live in Git; updating = git pull + tests + deploy.

GitHub

What it is: a website that hosts Git repositories.

Why: your repository is reachable from any machine; you can fork other people's projects.

Analogy: a cloud drive, but for code and with full history.

Common mistake: committing a secret to a public repo. The key is instantly compromised — revoke it.

In Wayan: the install runs straight from github.com/wayanonchain/wayan-claude-install.

Repository

What it is: a project folder managed by Git: the files plus their entire history.

Why: the unit of work: clone the repository — get the whole project.

Analogy: a binder containing every draft of every version of a document.

Common mistake: running install commands from outside the repository folder.

In Wayan: on the server the repository lives at /root/wayan-claude-install.

Commit

What it is: a recorded "snapshot" of changes with a note about what changed and why.

Why: every step of history has a name and can be reverted.

Analogy: a logbook entry: date, what was done, by whom.

Common mistake: one giant commit called "fixed everything". Small, meaningful commits are easier to revert.

In Wayan: "teaching" an agent is also commits: new rules and knowledge enter history.

Push / Pull

What it is: push — send your commits to GitHub; pull — fetch new ones to yourself.

Why: that's how your local copy and GitHub stay in sync.

Analogy: sending and receiving parcels between your desk and the warehouse.

Common mistake: forgetting to pull on the server and wondering why there are "no updates".

In Wayan: scripts/update.sh does the pull + tests for you.

Branch

What it is: a parallel line of history inside a repository.

Why: experiment without breaking the main version (main).

Analogy: a draft kept apart from the final copy: finish it, check it — then merge it in.

Common mistake: living in your own branch for months and accumulating conflicts with main.

In Wayan: installs and deploys run from the main branch.

Docker

What it is: running programs in isolated "containers" with their whole environment.

Why: the program brings everything it needs — no conflicts with the system.

Analogy: food in a lunchbox: reheat it anywhere, no dishes required.

Common mistake: exposing a container's port to the whole internet (0.0.0.0) instead of localhost.

In Wayan: OpenViking runs in Docker and listens only on 127.0.0.1:1933.

systemd

What it is: the Linux service manager: starts programs at boot and restarts crashed ones.

Why: the agents must survive a server reboot without your help.

Analogy: the building superintendent: turns the lights on, resets tripped breakers.

Common mistake: starting a program by hand in a terminal and closing it — the program dies with the session.

In Wayan: wayan-jupiter.service and wayan-uran.service are managed by systemd.

Service

What it is: a program under systemd's management: background, auto-started, observable.

Why: uniform commands: start / stop / restart / status + logs in journald.

Analogy: an appliance on a breaker-protected outlet: trips — the breaker brings it back.

Common mistake: editing the env file and forgetting systemctl restart — settings are read at startup.

In Wayan: systemctl status wayan-jupiter.service is the first thing to check when in doubt.

Environment variable

What it is: a named value a program reads at startup (KEY=value).

Why: settings and secrets live outside the code — code is public, values are private.

Analogy: sticky notes on the fridge for the housekeeper: instructions separate from the work itself.

Common mistake: quotes/spaces around the value (TOKEN = "abc") — write TOKEN=abc.

In Wayan: all agent settings live in /etc/wayan-jupiter.env and /etc/wayan-uran.env.

API key

What it is: a secret string by which a service recognizes you and charges your account.

Why: Groq and OpenAI only work with a key.

Analogy: a bank card: whoever holds it, pays with it. Not a "password" — a card.

Common mistake: pasting the key into a chat/screenshot/GitHub. Leaked — revoke immediately.

In Wayan: keys live only in env files (permissions 600) and ov.conf — never in Git.

Telegram bot

What it is: a program-account in Telegram, controlled by your server.

Why: a ready-made interface: chat, voice, files — no app development.

Analogy: your system's front desk: all communication goes through it.

Common mistake: using one token for both agents — Uran stops being an independent channel.

In Wayan: two bots: one for Jupiter, one for Uran, tokens in separate env files.

BotFather

What it is: Telegram's official bot that creates other bots.

Why: the only legitimate way to get a bot token.

Analogy: the passport office for bots.

Common mistake: falling for a fake "BotFather". The real one is @BotFather with a verified badge.

In Wayan: /newbot twice — once for Jupiter, once for Uran.

Claude Code

What it is: Anthropic's agentic environment: the Claude model + tools (terminal, files) in a CLI.

Why: this is the agents' brain — it doesn't just answer, it acts.

Analogy: not a "reference book" but an intern with computer access: reads, writes, runs.

Common mistake: logging in as root. The login must happen as the wayan user — services run as it.

In Wayan: the gateway calls claude -p for every task arriving from Telegram.

Groq

What it is: an ultra-fast inference service; we use its Whisper — speech recognition.

Why: voice notes and video audio become text in seconds.

Analogy: a stenographer who gets through an hour of audio in half a minute.

Common mistake: putting the Groq key in the wrong env (you need it in both, if both agents should hear).

In Wayan: GROQ_API_KEY in the env; the model is whisper-large-v3-turbo.

OpenAI embeddings

What it is: turning text into a vector of numbers where similar meaning = nearby vectors.

Why: that's how memory searches "similar in meaning" rather than word matches.

Analogy: coordinates on a "map of meanings": neighboring points are related thoughts.

Common mistake: not funding the OpenAI balance — memory silently fails to store.

In Wayan: OpenViking uses text-embedding-3-small + gpt-4o-mini.

OpenViking

What it is: a semantic memory server, running in Docker on your VPS.

Why: long-term memory: "find similar past cases" works through it.

Analogy: a librarian who remembers every book by its content, not by its spine.

Common mistake: exposing its port to the internet. It must listen on localhost only.

In Wayan: 127.0.0.1:1933, memory shared by Jupiter and Uran, data in ~/.openviking.

Semantic memory

What it is: memory that answers "what was that about?", not "where is this exact phrase?".

Why: the agent recalls what's relevant without re-reading every file every time.

Analogy: human memory: you remember the gist of a conversation, not the transcript.

Common mistake: dumping everything into memory — garbage in, garbage out.

In Wayan: short, important entries are stored; the source of truth stays in Markdown+Git.

Markdown

What it is: a text format with light markup: # heading, - list, **bold**.

Why: equally readable by a human, an agent, and Git; depends on no particular app.

Analogy: the pen and paper of the digital world: it will outlive every trendy app.

Common mistake: keeping knowledge in proprietary formats the agent can't read.

In Wayan: rules, skills, memory, transcripts — all of it is Markdown files.

Obsidian

What it is: a free app for working with a folder of Markdown notes (a "vault").

Why: the comfortable human interface for reading, editing, and linking the agents' knowledge.

Analogy: a reading room above the archive: same documents, nicer to work with.

Common mistake: thinking Obsidian "stores" the data. It only displays ordinary files.

In Wayan: open the knowledge folder as a vault — and see everything the agents have built.

ffmpeg

What it is: the universal command-line audio/video converter.

Why: it pulls keyframes out of videos so Claude can "see" them.

Analogy: a Swiss army knife for media: cut, compress, convert.

Common mistake: enabling visual video analysis without installing ffmpeg — the agent will honestly tell you.

In Wayan: VIDEO_VISUAL_ANALYSIS=true + ffmpeg = videos get analyzed visually too.

Webhook

What it is: event delivery where the service itself knocks on your address with an HTTP request.

Why: instant delivery, but it requires an open port — which is attack surface.

Analogy: a courier ringing your doorbell — convenient, but your door must face the street.

Common mistake: leaving the webhook port open to the whole internet.

In Wayan: the agents don't use webhooks (see polling). The legacy bot's public webhook port 8080 existed once — closed in phase M6.

Polling

What it is: the program itself periodically asks the server: "any new messages?"

Why: no ports need to be opened — outbound requests are safer than inbound ones.

Common mistake: assuming polling is "slow" — long polling delivers messages almost instantly.

Analogy: checking your own mailbox instead of giving the courier a key to your door.

In Wayan: both agents receive Telegram messages via long polling. Open ports: zero.

Firewall

What it is: a network connection filter: which ports are open, which are closed.

Why: everything that shouldn't be reachable from the internet must be closed.

Analogy: security at the entrance: only people on the list get in.

Common mistake: opening a port "just for a minute to test" and forgetting it.

In Wayan: ufw: essentially only SSH is open; the legacy bot's 8080 rule was removed.

No-clobber

What it is: a copy mode that "never overwrites what exists" (cp -n).

Why: re-running the installer never wipes your edited templates, rules, or memory.

Analogy: a mover who brings in the missing furniture but never touches what you've already arranged.

Common mistake: expecting a re-install to "update" a file you edited — no, your edits win.

In Wayan: the installer and WAYAN_PACK packs copy everything no-clobber only.

Deploy

What it is: delivering a new code version into production.

Why: so "updating the agents" is one verifiable procedure, not ad-hoc edits.

Analogy: printing a new edition of a book: proofread it — then replace it on the shelves.

Common mistake: "deploying" by editing a file directly on the server. That's exactly what drift is.

In Wayan: scripts/deploy-gateway.sh: backup → sync → verify → restart.

Rollback

What it is: returning to the previous working version after a failed update.

Why: any update can break — there must be a way back within minutes.

Analogy: "undo" in an editor, but for the whole server.

Common mistake: having no backup and fixing a broken system "forward", in a panic.

In Wayan: every deploy leaves /opt/wayan-*/gateway.bak.DATE — rollback = restore the folder and restart.

WAYAN_PACK

What it is: the environment variable that selects a profession pack at install time.

Why: one installer — different specializations: onchain, creator, devops, student, founder.

Analogy: a car trim level: same base, options for your scenario.

Common mistake: forgetting -E in sudo -E — sudo swallows the variable and the pack isn't applied.

In Wayan: WAYAN_PACK=creator sudo -E ./install.sh. Without the variable — a plain install.

Drift

What it is: divergence between the code in Git and the code actually running in production.

Why it matters: drift is the silent enemy: "it worked yesterday" and nobody knows why.

Analogy: the blueprint says one thing, but someone has already welded something else onto the machine.

Common mistake: "I'll quickly fix it on the server and move it to Git later" — and never moving it.

In Wayan: bash scripts/deploy-gateway.sh --check — a PASS/FAIL drift report (file names only).

Section 7

Three installation paths

Path A. From a Mac (or any computer) to your own VPS — recommended for beginners

Your computer is the remote control; everything installs onto the server. First check your tools (both ship with macOS; on Windows use PowerShell or WSL):

git --version
ssh -V

Connect to the server (the IP and password come from your VPS provider):

ssh root@SERVER_IP

To stop typing the password every time, create an SSH key. The key belongs to you (to your client computer) — not to Wayan and not to the server: the private half never leaves your laptop; only the public half is copied to the server.

ssh-keygen -t ed25519 -C "my-wayan-key"
ssh-copy-id root@SERVER_IP

Then add a convenient alias to ~/.ssh/config:

Host wayan-vps HostName SERVER_IP User root IdentityFile ~/.ssh/id_ed25519

From now on, connecting is just ssh wayan-vps.

Cursor / VS Code Remote-SSH. If you install the Remote-SSH extension (or use Cursor), you can open server files as if they were local — handy for editing templates and reading logs. It's optional: everything in this guide also works in a plain terminal.

Path B. Directly on the VPS

If your provider offers a web console (a terminal in the browser), you can skip SSH from your computer entirely: open the console in the provider's dashboard and run the same install commands from section 8, starting at step 4. Downside: a browser console is clumsy for longer work — path A is worth learning eventually anyway.

Path C. Fork and customize the repo — for developers

  1. Fork the wayanonchain/wayan-claude-install repository on GitHub.
  2. Rewrite the templates for yourself: templates/jupiter/CLAUDE.md, templates/uran/CLAUDE.md, USER.md, skills.
  3. Add or modify packs in packs/ (see section 9 and docs/PACKS.md).
  4. Install from your own fork: change REPO_URL in install.sh, or clone the fork and run locally.
  5. Never commit secrets: env files, tokens, ov.conf, accounts.md — all already covered by .gitignore; don't work around it.
Section 8

Step-by-step installation

18 steps from zero to a fully verified system. For each: the command, what it means, the expected output, and the common error. Don't rush: verify each step before moving to the next.

Step 1. GitHub account

What: sign up at github.com. Why: the system installs from here; later your knowledge base and your fork live here too.

Verify: you can open github.com/wayanonchain/wayan-claude-install and see the code. Common mistake: postponing it — without GitHub there's no path C and no knowledge export.

Step 2. Create a VPS

What: with any provider, create a server: Ubuntu 22.04 or 24.04, at least 1 GB, ideally 2 GB RAM. Write down the IP and the root password.

Why: this is the agents' home. Verify: the server shows "running" and you know its IP.

Common mistake: picking a different OS (Debian, CentOS) — the installer supports only Ubuntu 22.04/24.04 and will honestly refuse.

Step 3. Connect over SSH

ssh root@SERVER_IP

What: you're entering the server. On first connect, accept the fingerprint (yes), then enter the password.

Expected: a prompt like root@hostname:~# — commands now run on the server.

Common mistake: Permission denied — a typo in the password or IP; "connection timed out" — the server is still provisioning or the IP is wrong.

Step 4. Install Wayan Agents

Option 1 — one command (faster):

curl -fsSL https://raw.githubusercontent.com/wayanonchain/wayan-claude-install/main/install.sh | sudo bash

Option 2 — from a clone (you can read what you're running; required for WAYAN_PACK):

git clone https://github.com/wayanonchain/wayan-claude-install.git cd wayan-claude-install sudo ./install.sh

What happens: packages, Node.js, and Claude Code are installed; the wayan user is created; workspaces, the gateway, and systemd services are set up. The installer is idempotent — running it again breaks nothing and never overwrites your edits (no-clobber).

Expected: a green final banner "installation complete" with a NEXT STEPS section.

Common mistake: running without sudo → "This installer must run as root". Want a profession pack — read section 9 before this step.

Step 5. Create two bots with BotFather

What: in Telegram, open @BotFather → /newbot → a name and username for Jupiter; repeat for Uran. Save both tokens (format 123456:ABC-DEF...).

Why two: Uran is the independent channel; if Jupiter's bot is unreachable, you still have a line to the server.

Common mistake: swapping the tokens — "Jupiter" will answer in Uran's chat and vice versa.

Step 6. Put the tokens into the env files

sudoedit /etc/wayan-jupiter.env

Find the line and paste the token (no quotes, no spaces):

TELEGRAM_BOT_TOKEN=YOUR_BOT_TOKEN
sudoedit /etc/wayan-uran.env

Why: the gateway reads the token from the env at startup — that file is the only place it lives.

Common mistake: editing and forgetting to restart the service (step 12) — the env is read only at startup.

Step 7. Log in to Claude Code as the wayan user

sudo -u wayan -i claude

What: become the wayan user and run claude — complete the browser login flow (a Pro/Max subscription is required). Then leave claude and return to root: exit.

Why: the services run as wayan — a root login is invisible to them.

Expected: claude --version works; launching claude again doesn't ask for login.

Common mistake: logging in as root: the agents will reply with an error to every message.

Step 8. Add the Groq key (voice)

sudoedit /etc/wayan-jupiter.env
GROQ_API_KEY=YOUR_GROQ_KEY

Repeat for /etc/wayan-uran.env if Uran should understand voice too.

Why: without the key, voice messages are politely declined; text keeps working normally.

Common mistake: creating the key but never checking limits/balance at console.groq.com.

Step 9. OpenAI key for memory

What: create a key at platform.openai.com and fund the balance with at least $5 — with no funds, embeddings don't work and memory can't store anything.

Where it goes: into the OpenViking config /home/wayan/.openviking/ov.conf (next step) — not into the agents' env files, and certainly not into Git.

Common mistake: key exists, balance is $0 → memory_store fails with a cryptic quota error.

Step 10. Install and start OpenViking

Section 9

WAYAN_PACK: profession packs

A pack is a profession preset chosen at install time. Technically it's a set of Markdown files layered on top of the normal install: a role emphasis (PACK.md), a memory seed, task-routing rules, and a services map. Packs exist so the agent "thinks" in your profession from day one instead of being tuned for weeks.

Three pack guarantees. 1) It's Markdown only — no new services, keys, or permissions. 2) No-clobber: a pack never overwrites existing files — your edits always win. 3) Without WAYAN_PACK the install is identical to the plain one — packs change nothing until you ask for them.
sudo ./install.sh
WAYAN_PACK=onchain sudo -E ./install.sh
WAYAN_PACK=creator sudo -E ./install.sh
WAYAN_PACK=devops sudo -E ./install.sh
WAYAN_PACK=student sudo -E ./install.sh
WAYAN_PACK=founder sudo -E ./install.sh

The -E flag is required: it passes the variable through sudo. A typo in the pack name is safe — the installer refuses before changing anything and prints the list of available packs.

🪙 onchain — crypto analyst

Trader, analyst, alpha-channel owner. This is Wayan's "native" profile, formalized as a pack.

Jupiter: token and smart-money breakdowns, watch/avoid/deep-dive verdicts, threads from research.

Section 11

Where Obsidian fits

The simple idea: agents write Markdown — Obsidian shows it beautifully to a human. No magic: Obsidian just opens the same folder of files that the agents write and Git stores.

  • Obsidian is for humans. The agents don't need it: they read the files directly.
  • Git stores the same files with history; edit freely — changes get committed.
  • You review and edit notes, link them, prune what's stale.
  • Then — commit to Git: your edits become the official version of the knowledge.
Voice / video / PDF ↓ Markdown transcript (agent) ↓ You review and edit in Obsidian ↓ Approved knowledge → commit to Git ↓ OpenViking indexes → agents recall

How to connect it: install Obsidian → "Open folder as vault" → point it at the knowledge folder (a local clone of the repo, or a synced transcripts folder). Done: the agents' knowledge base is in front of you.

Section 12

How to teach the agents

The agents learn through a safe, human-in-the-loop cycle:

  1. You say: "remember: …" / "don't do that" / "fix this".
  2. Durable facts go into memory (memory_store / cold.md).
  3. Behavioral lessons get written into learnings/inbox/.
  4. On request, the agent proposes: "these lessons should become a rule/skill/memory entry" — as a proposal file, not an edit.
  5. You approve — and only then is the change applied.
  6. The change goes into Git as a commit — learning has history and a rollback.
Why so strict? No self-editing by default: an agent that rewrites its own rules quickly drifts away from your intent — so the permission profiles explicitly forbid the agents from editing their CLAUDE.md, rules, skills, and memory without approval. The agent proposes — the human decides: the same logic as code review. Secrets are never stored, no matter what you type in the chat. Rules and skills change carefully: in small, reversible, versioned steps.

Task queue and media behavior

The queue

  • Instant ack. Every message is confirmed immediately: "✅ Task received. Queue position: N" — you always know the task was accepted.
  • Per-chat queue. Tasks run one at a time, in order — feel free to send several in a row.
  • /queue — show the queue; /cancel — drop pending tasks.
  • Timeouts are not silence. If a task exceeds its limit, you get "⏱ … Moving on to the next task" and the queue continues.
  • The no-silence guarantee. Every outcome — answer, error, timeout — is reported explicitly. Silence = a reason to message Uran.
  • Unsupported files get a clear refusal, not a shrug.

Section 16

Updates: repo-first deployment

Why drift is dangerous. If code is edited directly in production, Git stops being the truth: the next deploy silently overwrites the edit, there's nothing to roll back to, and debugging becomes archaeology. So there is one rule: changes are born in Git and reach the server only through the scripts.

The correct update cycle

cd /root/wayan-claude-install
bash scripts/update.sh
sudo bash scripts/deploy-gateway.sh --dry-run
sudo bash scripts/deploy-gateway.sh --check
sudo bash scripts/deploy-gateway.sh --restart
CommandWhat it doesChanges anything?
update.shgit pull + the full test suite + a drift checkno repository only
--dry-runshows what a deploy would dono
--checka Git ↔ production drift report: PASS/FAIL, file names onlyno
--restartbackup → sync both /opt trees → import check → service restart → re-verifyyes this is the deploy

Backups and rollback

Every deploy automatically leaves a snapshot: /opt/wayan-*/gateway.bak.YYYYMMDD-HHMMSS. Rollback on trouble: stop the service → restore the folder from the backup → start → check status. By construction, the deploy script never touches env files, secrets, OpenViking, or the workspaces.

⚠️ Never edit directly: /opt/wayan-*/gateway (code travels only via deploy), the backup contents, and don't "quick-fix on the server" anything that belongs in Git. Env files are the one thing edited directly on the server (they're secrets; they must not enter Git).
Section 17

Safety and permissions

MechanismWhat it is and why
Role-based permissionsEach agent has its own .claude/settings.json: read-only diagnostics and research are allowed; dangerous things are gated. Jupiter is a worker without admin rights; Uran can additionally restart the wayan services (and only them) via a narrow sudoers file.
Gates on dangerous opsgit push, ssh, rm, chmod, stopping services, apt, reboot — all require explicit approval. A model mistake must not be able to level the server.
No-self-edit deniesDeny rules stop an agent from editing its own CLAUDE.md, rules, skills, memory. Learning happens only through proposals and your approval (section 12).
Secret protectionTokens and keys live only in /etc/wayan-*.env and ov.conf with 600 permissions; .gitignore blocks their way into Git; agents are forbidden from echoing values into logs or chats.
OpenViking is localhost-onlyMemory listens on 127.0.0.1:1933 and is unreachable from the internet: it holds your knowledge, and it must not face outward.
Port 8080 closed (M6, phase 1)The legacy bot's public webhook port was closed: the bot now binds to localhost and the firewall rule was removed. Fewer open ports — smaller attack surface.
.env at 600Env file permissions were hardened: only root and the services can read them. No other user/process sees the tokens.
Rootless migration (M6, phase 2)The legacy wayan_pirat_bot still runs as root — the plan to move it to an unprivileged user is written but not yet applied (see docs/M6_WAYAN_BOT_ROOTLESS_PLAN.md). Know this limitation exists.
Section 18

When something breaks

19 typical problems. Open your card: symptom → likely cause → fix → check command.

SSH: password rejected / key doesn't work

Symptom: Permission denied (publickey,password). Cause: a typo, wrong IP, or the key wasn't copied to the server.

Fix: verify the IP in the provider dashboard; the password is typed blind — that's normal; for keys: ssh-copy-id root@SERVER_IP.

ssh -v root@SERVER_IP
git pull fails

Symptom: conflict / divergent branches / local changes would be overwritten. Cause: someone hand-edited the clone on the server.

Fix: look at what changed; commit what's valuable, drop the junk.

cd /root/wayan-claude-install && git status && git stash && git pull
install.sh: permission denied / must run as root

Symptom: refusal at launch. Cause: no sudo, or the file isn't executable.

Fix: run with sudo; if needed, chmod +x install.sh or sudo bash install.sh.

sudo bash install.sh
Telegram bot is silent (token)

Symptom: the bot doesn't respond at all. Cause: an empty/wrong token, swapped tokens, or the service wasn't restarted after editing the env.

Fix: compare the token against BotFather, fix the env, restart the service.

sudo systemctl restart wayan-jupiter.service && journalctl -u wayan-jupiter.service -e
Claude not logged in

Symptom: the agent errors on every message; auth errors in the logs. Cause: the login was never done, or was done as root.

Fix: log in specifically as wayan (step 7) and restart the services.

sudo -u wayan -i claude
Voice isn't transcribed (Groq)

Symptom: "voice is disabled" or an error on voice notes. Cause: no GROQ_API_KEY, an invalid key, or no restart.

Fix: check the key in the env and the balance/limits at console.groq.com, restart the service.

grep -c GROQ_API_KEY /etc/wayan-jupiter.env && sudo systemctl restart wayan-jupiter.service
OpenViking health doesn't answer

Symptom: curl to /health hangs or errors. Cause: the container isn't running / Docker didn't start.

Fix: bring compose up, read the container logs.

cd /home/wayan/.openviking && docker compose up -d && docker compose logs --tail 50
memory_store fails

Symptom: the agent says it couldn't store the memory. Cause #1: the OpenAI balance is empty (needs ≥ $5); or OpenViking isn't running; or the key in ov.conf is wrong.

Fix: fund the OpenAI balance; check /health; check ov.conf permissions (600, wayan:wayan).

curl -s http://127.0.0.1:1933/health
Docker not installed

Symptom: docker: command not found. Cause: Docker isn't part of the base Wayan installer — it's needed only for OpenViking.

Fix: install Docker Engine + the compose plugin via the official docs.docker.com guide for Ubuntu, then return to step 10.

docker --version && docker compose version
ffmpeg missing

Symptom: the agent says visual analysis is unavailable and reviewed audio only. Cause: ffmpeg isn't installed.

Fix: install it and restart the service.

sudo apt-get install -y ffmpeg && sudo systemctl restart wayan-jupiter.service
Videos analyzed by audio only

Symptom: no visual part in the reply. Cause: VIDEO_VISUAL_ANALYSIS is off (the default is false).

Fix: enable the flag in the env of the agent(s) and restart.

sudoedit /etc/wayan-jupiter.env # VIDEO_VISUAL_ANALYSIS=true
Video too large

Symptom: the agent says the file exceeds the limit. Cause: Telegram doesn't let bots download files over ~20 MB.

Fix: send a direct link to the file (a URL to an .mp4 etc.) — the gateway downloads it itself with size and disk checks.

Queue stuck

Symptom: "Queue position: 3" and silence beyond the timeout. Cause: a long task or a hung Claude process.

Fix: /queue — inspect; /cancel — clear; if that fails — ask Uran "why is Jupiter not responding" or restart the service.

sudo systemctl restart wayan-jupiter.service
Agent says: permission denied

Symptom: the agent reports that an action is blocked by permissions. Cause: the role permission profile is doing its job — the action is on the dangerous list.

Fix: most of the time this is exactly as designed. If the action is genuinely needed — consciously extend the agent's .claude/settings.json (see docs/PERMISSIONS.md), don't switch the protection off wholesale.

Low RAM / server swapping

Symptom: everything is slow; the health report shows swap. Cause: 1 GB RAM is tight for the agents + OpenViking.

Fix: upgrade the VPS to 2 GB — the cheapest fix for this system's performance problems.

free -h && swapon --show
Drift: production doesn't match Git

Symptom: --check shows FAIL and file names. Cause: someone edited /opt directly, or a deploy didn't finish.

Fix: first figure out which version is correct. If the edit is valuable — move it into the repository as a commit, then deploy. Don't overwrite blindly.

cd /root/wayan-claude-install && bash scripts/deploy-gateway.sh --check
Service crash-looping

Symptom: status shows activating/auto-restart; the bot flickers alive and dead. Cause: a config error (broken env) or a code error.

Fix: read the last journal lines — the exact cause is there; fix the cause, not the symptom.

journalctl -u wayan-jupiter.service -n 50 --no-pager
Ports and firewall: something "unreachable"

Symptom: it feels like you need an open port, or that something is exposed. Cause: webhook vs polling confusion.

Fix: remember: Wayan agents need no inbound ports (Telegram works via polling, OpenViking is localhost). Only SSH should face the internet.

sudo ufw status && ss -tlnp | grep -v 127.0.0.1
Accidentally pasted the prompt together with the command

Symptom: errors like root@server:~#: command not found or $: command not found. Cause: the line was copied together with the shell prompt ($, #, root@…).

Fix: copy only the command itself. In this guide, the "copy" button gives you the clean command with no prompts.

The repo as a template: build your own system

Wayan Agents is an open template — a reusable framework anyone can take as the foundation of their own agent system:

  1. Fork the repository on GitHub — you get your own copy with history.
  2. Pick a pack (WAYAN_PACK) as the starting specialization — or write your own pack following docs/PACKS.md.
  3. Customize the templates: templates/*/CLAUDE.md and USER.md — your agents' names, roles, and tone.
  4. Customize the skills in skills/ — playbooks for your tasks.
  5. Customize the orchestration: rules, the services map, the memory seed in orchestration/.
  6. Secrets stay local only: env files, ov.conf, accounts.md are never committed (already enforced by .gitignore).
  7. Git is for documents and code — not for runtime secrets and not for raw media.
  8. Obsidian is for knowledge: your Markdown base grows with the agents and remains yours.
  9. The result is your own brand and agent system: "Jupiter and Uran" easily become any characters of your world.
